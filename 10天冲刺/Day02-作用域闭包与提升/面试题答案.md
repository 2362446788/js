# Day 2 面试题答案

## 1. 什么是执行上下文？函数执行时到底发生了什么？

标准回答：

执行上下文我是这么理解的，在浏览器中，去请求我们的js代码，然后把js代码转成浏览器引擎能够识别的ast语法树，要执行的时候，就会开辟一块栈内存空间来供代码执行，这块空间叫做执行环境栈，这个时候代码就可以放到执行环境栈中执行，首先是全局的代码，会形成一个执行上下文，叫做全局执行下上文，除了全局执行上下文，还有函数执行上下文和块级执行上下文，这些执行上下文可以理解为“代码运行时的环境信息集合”，里面会记录当前代码执行需要的变量、函数、`this`、作用域链等信息。

JavaScript 常见有三类执行上下文：

- 全局执行上下文
- 函数执行上下文
- `eval` 执行上下文

函数执行大致分两阶段：

1. 创建阶段
   - 确定作用域链
2. 执行阶段
   - 绑定 `this`
   - 初始化 arguments
   - 行参赋值
   - 变量提升
   - 逐行执行代码
   - 变量赋值
   - 函数调用

## 2. 什么是作用域？什么是作用域链？

标准回答：

作用域是变量和函数可被访问的范围。JavaScript 主要有：

- 全局作用域
- 函数作用域
- 块级作用域

作用域链是：当前作用域查找不到变量时，沿着外层词法环境逐级向上查找，直到全局作用域为止。

## 3. JavaScript 为什么是词法作用域？

标准回答：

因为变量可访问范围在“定义时”就决定了，而不是“调用时”决定的。

也就是说函数能访问哪些变量，取决于它写在哪里，不取决于它在哪里被调用。

## 4. `var`、`let`、`const` 的区别是什么？

标准回答：

1. let不存在变量提升,不允许在定义之前使用
2. let不允许重复声明「不论当前上下文中，基于何种方式声明过这个变量，都不允许基于let再次声明」
3. 在全局上下文中，基于var/function声明的变量，是给window(GO)设置的全局属性；基于let/const声明的变量是放在VO(G)中的，和GO没有任何的关系
4. let会产生块级上下文
5. let的暂时性死区问题

/* =====let && const 的区别 =====*/
// const声明的是常量，这句话是不准确的，他声明的函数变量；基于const声明的变量，首先必须赋值初始值，而且一但和某个值关联，后期不允许更改其指针指向（也就是不能重新赋值为其他的值）;

- `var`
  - 函数作用域
  - 允许重复声明
  - 有变量提升
  - 全局下可能挂到 `window`
- `let`
  - 块级作用域
  - 不允许重复声明
  - 有暂时性死区
- `const`
  - 和 `let` 一样是块级作用域
  - 声明时必须初始化
  - 不能改绑定，但如果是对象，内部属性仍可变

## 5. 什么是暂时性死区？

标准回答：

从作用域开始到变量声明完成之前，这段区域里访问 `let` / `const` 声明的变量会报错，这就是暂时性死区。

它的意义是避免变量在声明前被误用。

## 6. 什么是变量提升？函数提升和变量提升的差异是什么？

标准回答：

变量提升不是“代码真的移动了”，而是创建阶段会先收集声明信息。

- `var`：会提升声明，初始值是 `undefined`
- 函数声明：会连同函数体一起提前可用
- `let/const`：也会被注册，但声明前不可访问

函数声明优先级高于 `var` 声明。

## 7. 为什么函数里声明的变量能访问外层变量？

标准回答：

函数能访问外层变量，是因为函数在定义时就记住了它所在的词法环境，函数执行时会形成作用域链。当前作用域找不到变量时，会沿着外层词法环境向上查找，所以函数内部能访问定义它时所在外层作用域的变量。

## 8. 什么是闭包？请给一个项目中的实际例子。

标准回答：

~~闭包的本质不是“函数嵌套函数”，而是：函数在定义时所在的词法环境，被函数在外部继续访问时保留下来。~~

对于闭包我是这么理解的，在一个函数A中，定义了一个函数B，函数B在定义时通过内部 [[Environment]] 属性捕获了当前词法环境，即函数A的作用域；当函数A执行，函数B作为值被传递到外部作用域并保持引用时，即使函数A执行完毕，这个被捕获的词法环境仍然无法被 GC 回收，从而实现了函数B能够持久访问函数A中的变量。

项目例子：

- 防抖函数里保留 `timer`
- 模块化封装里保留私有状态
- 柯里化函数里保留前面传入的参数

## 9. 闭包有哪些优点和缺点？

标准回答：

优点：

- 封装私有变量
- 延长变量生命周期
- 实现缓存、柯里化、防抖节流

缺点：

- 如果不注意引用释放，可能增加内存占用
- 调试时链路更复杂

## 10. 闭包为什么可能导致内存占用增加？

标准回答：

因为闭包会让外部函数中的某些变量在函数执行结束后仍然被引用，GC 不能立即回收它们。

注意这里是“可能”，不是“必然”。只要引用还在，变量就不能回收；引用断开后，仍然可以被回收。

## 11. 垃圾回收大致怎么做？你知道哪些常见策略？

标准回答：

浏览器和 JS 引擎会自动做垃圾回收。核心原则是：找出不再可达的对象并回收。

常见理解方式：

- 标记清除：从根对象出发标记可达对象，未标记的回收
- 引用计数：早期方案，循环引用下有缺陷

现代引擎主要用标记清除及其优化版本。

## 12. 为什么说“会不会泄漏”取决于引用是否还存在？

标准回答：

因为垃圾回收判断的核心不是“这个对象看起来有没有用”，而是“它是否还能从根对象访问到”。

只要引用链还在，这个对象就是可达的，就不会被回收。

## 13. 如何实现防抖？适合什么场景？

标准回答：

防抖的核心是：频繁触发时只认最后一次，或者只认第一次。

典型实现思路：

```js
function debounce(fn, wait) {
  let timer = null;
  return function (...args) {
    clearTimeout(timer);
    timer = setTimeout(() => fn.apply(this, args), wait);
  };
}
```

适用场景：

- 输入框搜索
- resize
- 频繁点击提交

## 14. 如何实现节流？适合什么场景？

标准回答：

节流的核心是：高频触发时按固定频率执行。

常见实现思路：

- 时间戳版
- 定时器版

适合场景：

- scroll
- mousemove
- 拖拽

## 15. 什么是内存泄漏？

标准回答：

内存泄漏是指：本来已经不再需要的对象，仍然被引用，导致垃圾回收器无法释放，长期积累后会造成内存持续增长。

常见场景：

- 未清理的定时器
- 未移除的事件监听
- 闭包长期持有大对象
- 全局变量滥用
- DOM 删除后仍被 JS 引用

## 18. 作用域链如何延长？

标准回答：

严格说是“某些机制会让变量查找路径变复杂或延续”。

常见包括：

- 闭包：让外层词法环境在函数外继续可访问
- `with`：把对象临时加到查找链前面，不推荐使用
- `try...catch`：`catch` 会引入自己的作用域绑定

面试里最重要的是讲清闭包这一点。
