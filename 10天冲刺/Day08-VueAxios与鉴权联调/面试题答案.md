# Day 8 面试题答案

## 1. 你在项目里是怎么封装请求库的？

标准回答模板：

“我一般会基于 Axios 做二次封装，主要做 5 件事：统一 `baseURL`、超时、请求头处理、请求拦截器、响应拦截器。这样业务层只关心接口调用，不用每次重复写 token、错误提示、状态码处理。对于项目里共性的逻辑，比如登录失效、请求格式转换、统一 loading、错误上报，也都放在这一层做。”

## 2. Axios 请求拦截器和响应拦截器分别适合做什么？

标准回答：

- 请求拦截器：
  - 注入 token
  - 统一设置请求头
  - 处理防重复提交
  - 序列化参数
- 响应拦截器：
  - 统一解包数据
  - 统一错误处理
  - 登录失效跳转
  - 接口异常上报

## 3. 为什么要统一做错误处理？

标准回答：

因为如果每个页面都自己处理错误：

- 代码重复
- 体验不统一
- 容易漏处理

统一错误处理能保证：

- 提示风格一致
- 认证失效行为一致
- 监控和日志统一收集

## 4. Token 一般放在哪里？各自有什么风险？

标准回答：

常见位置：

- `localStorage`
- `sessionStorage`
- Cookie
- 内存

风险：

- `localStorage/sessionStorage`：容易被 XSS 读到
- Cookie：如果配置不当可能有 CSRF 风险，但可借助 `HttpOnly` 提高安全性
- 内存：刷新会丢失

所以实际方案要结合业务安全要求选。

## 5. Token 过期后你怎么处理？

标准回答：

常见流程：

1. 响应拦截器识别鉴权失败状态码
2. 清理本地登录态
3. 跳登录页或触发刷新 token 流程
4. 避免多个并发请求同时重复跳转

## 6. 为什么有的项目要做请求取消或防重复提交？

标准回答：

因为高频操作可能带来两个问题：

- 用户多次点击导致重复提交
- 旧请求返回覆盖新请求结果

所以会做：

- 请求取消
- 按钮节流 / 防抖
- 相同请求合并

## 7. 为什么前端开发时常用代理转发？

标准回答：

主要是为了解决本地开发时的跨域问题，同时保持接口路径统一，方便开发环境和生产环境切换。

## 8. Vue 生命周期你最常用哪几个？

标准回答：

如果是 Vue 3 Composition API，常用的是：

- `onMounted`
- `onUpdated`
- `onUnmounted`

如果面试官接受 Vue 2 说法，也可以说：

- `created`
- `mounted`
- `beforeDestroy/destroyed`

## 9. `computed` 和 `watch` 的区别是什么？

标准回答：

- `computed` 适合派生值，本质是基于依赖缓存的计算结果
- `watch` 适合副作用，比如请求、日志、同步外部状态

## 10. `watch` 适合做什么，`computed` 适合做什么？

标准回答：

- `computed`
  - 表单展示字段拼接
  - 筛选后的列表
  - 基于状态推导 UI
- `watch`
  - 监听路由变化发请求
  - 监听输入变化做异步搜索
  - 同步某些数据到本地存储

## 11. `nextTick` 的作用是什么？

标准回答：

`nextTick` 用于在 DOM 更新完成后执行回调。因为 Vue 的响应式更新通常是异步批处理的，修改数据后 DOM 不会立刻同步更新，所以如果你要在更新后拿最新 DOM，就需要 `nextTick`。

## 12. Vue 组件通信有哪些方式？

标准回答：

常见方式：

- `props` / `emit`
- `v-model`
- `provide/inject`
- 状态管理（Pinia/Vuex）
- 事件总线（不推荐滥用）
- refs / expose

## 13. 父子、兄弟、跨层级组件通信你分别怎么做？

标准回答：

- 父子：`props` + `emit`
- 兄弟：共同父组件中转，或状态管理
- 跨层级：`provide/inject` 或状态管理

## 14. 你如何描述一次完整的登录流程？

标准回答模板：

“用户提交账号密码后，前端调用登录接口，服务端校验成功返回 token。前端保存 token，然后后续请求在拦截器里统一带上。页面初始化时会检查登录态并拉取用户信息。若接口返回 token 失效，就在响应拦截器里统一处理，比如清空状态并跳转登录页。如果项目要求更高，也会接入 refresh token 机制。”

## 15. 你如何描述一次前后端联调中的问题排查过程？

标准回答模板：

“我一般按 4 步排查：先看请求有没有发出去，再看请求参数和请求头是否正确，然后看服务端返回结构和状态码，最后再看前端解包和渲染逻辑。如果是跨域或代理问题，会先看浏览器控制台和 Network 面板；如果是鉴权问题，会重点看 token 是否注入、是否过期、后端返回码是否符合约定。排查过程中我会尽量把问题定位到‘前端参数问题 / 网关问题 / 服务端逻辑问题 / 数据格式约定问题’中的一类，这样联调效率更高。”
